## 1. 프로젝트 한 줄 요약

웹캠으로 손 랜드마크를 추출하고, SVM 모델로 정적 수화 자음/모음 손모양을 분류한 뒤,
분류된 자모를 한글 텍스트로 조합해 실시간 자막 형태로 보여주는 수어 인식 MVP입니다.

현재 프로젝트는 정적 수어를 대상으로 합니다. 동적 수어는 제외했습니다.

## 2. 문제 정의

### 해결하려는 문제

수어 사용자와 비수어 사용자가 대화할 때, 한쪽이 수어를 모르면 즉시 의사소통이 어렵습니다.
특히 일상 대화, 학교, 공공기관, 병원, 민원 상황처럼 빠르게 의미를 전달해야 하는 상황에서
수어를 텍스트나 음성으로 변환하는 보조 도구가 필요합니다.

이 프로젝트는 그중 1차 목표를 다음과 같이 설정했습니다.

```text
수어 사용자의 정적 손모양 입력
-> AI 모델로 자음/모음 분류
-> 한글 텍스트 조합
-> 실시간 자막 표시
-> 비수어 사용자와의 소통 보조
```

### 왜 ML 문제인가?

손모양 이미지는 사람마다 크기, 위치, 각도, 카메라 거리, 조명 조건이 다릅니다. 단순히 픽셀이나
고정 좌표 규칙으로 구분하기 어렵습니다. 그래서 MediaPipe Hands로 손의 주요 관절 위치를 추출한 뒤,
이를 특징 벡터로 변환하고 ML 분류 모델이 라벨을 예측하도록 설계했습니다.

정리하면 문제를 다음 ML 문제로 바꿨습니다.

```text
입력: 한 손의 21개 손 랜드마크에서 계산한 83차원 특징 벡터
출력: 정적 수화 라벨 ID
문제 유형: 다중 클래스 분류
모델: SVM classifier
```

### 왜 정적 수어부터 했는가?

동적 수어는 시간에 따른 손의 이동 궤적과 순서가 중요합니다. 즉 단일 프레임 분류가 아니라
연속 프레임 시퀀스 모델링이 필요합니다. 프로젝트 1차 MVP에서는 인식 파이프라인을 안정화하는 것이
우선이므로, 움직임보다 손모양 자체가 중요한 정적 자모 분류부터 구현했습니다.

PPT 설명 문장:

> 본 프로젝트는 동적 수어 전체를 한 번에 해결하기보다, 먼저 정적 자모 인식과 한글 조합 흐름을
> 안정화하는 것을 목표로 했습니다. 이를 통해 수어 인식 시스템의 데이터 수집, 모델 학습, 실시간
> 추론, 자막 출력까지 전체 파이프라인을 검증했습니다.

## 6. 라벨 설계

### 학습 대상 라벨

학습 대상은 자음/모음/DELETE 총 32개입니다.

| 구분 | 라벨 |
| --- | --- |
| 자음 | ㄱ, ㄴ, ㄷ, ㄹ, ㅁ, ㅂ, ㅅ, ㅇ, ㅈ, ㅊ, ㅋ, ㅌ, ㅍ, ㅎ |
| 모음 | ㅏ, ㅐ, ㅑ, ㅒ, ㅓ, ㅔ, ㅕ, ㅖ, ㅗ, ㅚ, ㅛ, ㅜ, ㅟ, ㅠ, ㅡ, ㅢ, ㅣ |
| 제어 | DELETE |

### 직접 학습하지 않은 라벨

쌍자음과 일부 겹모음은 별도 손모양으로 학습하지 않고 조합 규칙으로 처리했습니다.

| 구분 | 직접 학습하지 않은 라벨 | 처리 방식 |
| --- | --- | --- |
| 쌍자음 | ㄲ, ㄸ, ㅃ, ㅆ, ㅉ | 기본 자음 반복 입력으로 조합 |
| 겹모음 | ㅘ, ㅙ, ㅝ, ㅞ | 기본 모음 조합으로 처리 |

예시:

```text
ㄱ + ㄱ -> ㄲ
ㅅ + ㅅ -> ㅆ
ㅗ + ㅏ -> ㅘ
ㅗ + ㅐ -> ㅙ
ㅜ + ㅓ -> ㅝ
ㅜ + ㅔ -> ㅞ
```

### START / END 제어

`START`와 `END`는 SVM 학습 대상이 아니라 양손 규칙 기반 제어로 처리합니다.

| 제어 | 의미 | 처리 방식 |
| --- | --- | --- |
| START | 입력 시작 | 양손 펼친손 규칙 |
| END | 입력 종료 | 양손 주먹 규칙 |
| DELETE | 마지막 자모 삭제 | 오른손 정적 손모양을 SVM으로 분류 |

START/END를 모델 분류가 아니라 규칙 기반으로 둔 이유는 입력 모드 제어가 잘못 분류되면
전체 문장 입력 흐름이 흔들리기 때문입니다. 제어 동작은 명확하고 안정적인 규칙으로 분리했습니다.

## 7. 특징 추출

### 입력 데이터

MediaPipe Hands는 손 하나에 대해 21개의 랜드마크를 제공합니다.

각 랜드마크는 다음 좌표를 포함합니다.

```text
x, y, z
```

### 최종 특징 벡터

모델에 들어가는 입력은 원본 이미지가 아니라 83차원 특징 벡터입니다.

| 특징 종류 | 개수 | 설명 |
| --- | ---: | --- |
| 관절 각도 | 15 | 손가락 관절의 굽힘 정도 |
| 손목 기준 정규화 좌표 | 63 | 21개 랜드마크 x/y/z 좌표를 손목 기준으로 정규화 |
| 손끝 거리 | 5 | 손목 또는 기준점 대비 손끝 위치 정보 |
| 합계 | 83 | SVM 입력 벡터 |

### 왜 이미지 전체를 쓰지 않았는가?

이미지 전체를 사용하면 배경, 조명, 얼굴, 옷, 카메라 해상도 같은 불필요한 정보가 많이 들어갑니다.
반면 MediaPipe 랜드마크를 사용하면 손의 구조 정보만 남길 수 있습니다.

PPT 설명 문장:

> 본 프로젝트는 원본 이미지를 직접 학습하지 않고 손의 21개 랜드마크에서 계산한 83차원 특징을
> 사용했습니다. 이렇게 하면 배경이나 조명보다 손가락의 상대적 위치와 각도에 집중할 수 있어
> 17,184프레임 규모의 데이터로도 SVM 학습과 실시간 분류를 수행할 수 있습니다.

## 8. 모델 선택

### 사용 모델

```text
StandardScaler + SVC
```

모델 상세:

| 항목 | 값 |
| --- | --- |
| 모델 타입 | scikit-learn SVM |
| 커널 | RBF |
| C | 10.0 |
| gamma | scale |
| class_weight | balanced |
| probability | true |
| random_state | 42 |
| 입력 차원 | 83 |
| 출력 클래스 | 32 |

### 왜 SVM인가?

SVM은 이미지 원본보다 차원이 낮고 구조화된 특징 벡터를 분류할 때 적합한 모델입니다. 이번 프로젝트는
원본 이미지를 사용하지 않고 83차원 특징 벡터를 사용했습니다. 또한 학습 대상이 32개 라벨이고,
총 데이터가 17,184프레임 규모이므로, 수십만 장 이상의 이미지가 필요한 대규모 CNN보다 SVM이 MVP에
적합했습니다.

SVM 선택 이유:

- 17,184프레임 규모의 데이터셋에서도 학습 가능
- 원본 이미지가 아니라 83차원 특징 벡터를 사용하므로 SVM 입력 구조와 잘 맞음
- 평균 SVM 예측 시간이 0.5294ms/frame으로 30FPS 프레임 예산 33.3ms의 약 1.6%
- 모델 구조가 발표에서 설명하기 쉬움
- FastAPI 실시간 추론에서 모델 예측 단계의 계산 부담이 작음

딥러닝을 쓰지 않은 이유:

- CNN은 보통 원본 이미지와 더 많은 학습 데이터가 필요함
- 동적 수어까지 포함하려면 LSTM/Transformer 등 시퀀스 모델이 필요함
- 현재 MVP 목표는 모델 복잡도보다 전체 파이프라인 완성도가 더 중요함

## 9. 학습 및 평가 방식

### 데이터 분할

학습 코드는 라벨별로 stratified split을 수행합니다.

```text
라벨별 전체 프레임: 537
라벨별 테스트 프레임: 108
테스트 비율: 약 20%
전체 테스트 프레임: 3,456
```

라벨별로 동일한 비율로 테스트 데이터를 분리했기 때문에 특정 라벨만 평가에 많이 들어가는 문제를
줄였습니다.

### 평가 지표

필수 성능 지표:

- Accuracy
- Macro F1-score
- Confusion Matrix
- 라벨별 Precision / Recall / F1-score

추가 성능 지표:

- 평균 신뢰도
- 평균 추론 시간

아직 별도 정량 리포트가 없는 항목:

- 실제 웹캠 테스트 정확도

실제 웹캠 테스트 정확도는 사람별 조명, 카메라 위치, 손 크기, 배경에 영향을 받기 때문에

### 지표 해석

Accuracy `0.9980`은 테스트 프레임 3,456개 중 3,449개를 올바르게 분류했고,

Macro F1-score `0.9980`은 32개 라벨의 F1-score를 라벨별로 계산한 뒤 평균낸 값입니다.
전체 Accuracy만 높으면 특정 라벨에 성능이 치우쳤을 가능성이 있지만, Macro F1-score도 `0.9980`으로
Accuracy와 거의 같기 때문에 특정 라벨만 잘 맞춘 것이 아니라 라벨 전반에서 균형 있게 맞췄다고
해석할 수 있습니다.

Mean confidence `0.9805`는 모델이 예측할 때 평균적으로 약 `98.05%`의 확률값을 가진 라벨을 선택했다는
의미입니다. 웹 UI의 초록색 기준은 `80% 이상`이므로, 테스트 평균 confidence는 UI의 초록 기준보다
약 `18.05%p` 높습니다. 다만 confidence가 높다고 실제 환경에서 항상 맞는 것은 아니므로, 웹 UI에서는
안정화 판단과 confidence 색상 표시를 함께 사용합니다.

Average inference time `0.5294 ms/frame`은 SVM 예측 한 번에 평균 약 0.53ms가 걸렸다는 의미입니다.
일반적인 실시간 웹캠 기준을 30FPS로 잡으면 한 프레임에 사용할 수 있는 시간은 약 `33.3ms`입니다.
SVM 예측 시간 `0.5294ms`는 이 33.3ms 예산의 약 `1.6%`입니다. 60FPS 기준 프레임 예산 `16.7ms`와
비교해도 약 `3.2%` 수준입니다. 그래서 모델 예측 단계 자체는 실시간 처리 병목이 아니라고 설명할 수
있습니다.

단, 실제 웹 화면의 전체 지연은 SVM 예측만이 아니라 카메라 프레임 획득, MediaPipe Hands 랜드마크 추출,
브라우저 렌더링, HTTP 요청까지 포함합니다. 따라서 `0.5294ms/frame`은 전체 서비스 지연이 아니라
SVM 분류 단계만의 추론 시간입니다.

## 12. Confusion Matrix 해석

Confusion Matrix는 실제 라벨과 예측 라벨을 비교한 표입니다. 대각선 값이 클수록 정확하게 맞춘 것이고,
대각선 밖의 값은 오분류입니다.

현재 결과는 대각선 정분류가 3,449프레임이고 대각선 밖 오분류가 7프레임입니다.
전체 테스트 프레임 3,456개 기준 오분류율은 약 `0.20%`입니다.

주요 오분류:

| 실제 라벨 | 예측 라벨 | 프레임 수 |
| --- | --- | ---: |
| ㅔ | ㅖ | 1 |
| ㅔ | ㅕ | 1 |
| ㅔ | ㅋ | 1 |
| ㅋ | ㅈ | 1 |
| ㅈ | ㅊ | 1 |
| ㄱ | ㅜ | 1 |
| ㄱ | ㅈ | 1 |

해석:

- 전체 3,456개 테스트 프레임 중 오분류가 7개입니다. 즉 정분류는 3,449개입니다.
- `ㅔ`는 총 108개 테스트 프레임 중 105개를 맞추고 3개를 놓쳤습니다.
- `ㅔ`의 recall은 `0.9722`로, 108개 중 105개를 맞췄다는 의미입니다.
- 일부 손모양은 손가락 위치가 비슷하거나 촬영 각도에 따라 랜드마크 패턴이 가까워질 수 있습니다.
- 실시간 웹캠에서는 프레임 하나의 예측만 사용하지 않고 안정화 로직을 사용하므로, 단일 프레임 오분류가
  곧바로 입력 확정으로 이어지지 않도록 설계했습니다.

PPT 설명 문장:

> Confusion Matrix를 보면 대부분의 값이 대각선에 몰려 있어 라벨별 분류가 안정적으로 이루어졌습니다.
> 전체 테스트 프레임 3,456개 중 오분류는 7개로 오분류율은 약 0.20%였습니다. 다만 ㅔ, ㄱ, ㅈ, ㅋ처럼
> 일부 라벨에서 소수 오분류가 발생했으며, 실제 서비스에서는 안정화 로직을 통해 이런 단일 프레임 오차가
> 바로 최종 입력으로 확정되지 않도록 처리했습니다.

## 13. 실시간 안정화 로직

실시간 웹캠은 매 프레임마다 예측값이 조금씩 흔들릴 수 있습니다. 따라서 예측 한 번만 보고 바로 문자를
입력하면 오입력이 많아질 수 있습니다.

이를 막기 위해 안정화 판단을 사용합니다.

현재 설정:

```text
STABILITY_WINDOW = 7
MIN_STABLE_RATIO = 0.7
MIN_CONFIDENCE = 0.75
```

의미:

- 최근 7개 관측값을 봄
- 같은 라벨이 최소 70% 이상 반복되어야 함
- 평균 confidence가 75% 이상이어야 함
- 조건을 만족한 라벨만 실제 입력 이벤트로 처리

발표 설명:

> 모델은 매 프레임마다 예측하지만, 실제 입력은 바로 확정하지 않았습니다. 최근 7프레임 중 같은 라벨이
> 충분히 반복되고 confidence가 일정 기준 이상일 때만 자모로 저장했습니다. 이 방식으로 손의 흔들림이나
> 순간적인 오분류가 최종 텍스트에 반영되는 문제를 줄였습니다.

## 14. 한글 조합 방식

모델은 완성된 글자를 직접 예측하지 않고 자음과 모음을 예측합니다. 이후 조합 로직이 자모를 한글 글자로
변환합니다.

예시:

```text
ㄱ + ㅏ -> 가
ㄴ + ㅏ -> 나
ㅅ + ㅅ + ㅏ -> 싸
ㅇ + ㅗ + ㅏ -> 와
ㅇ + ㅜ + ㅔ -> 웨
```

이 방식의 장점:

- 모든 완성형 글자를 직접 학습할 필요가 없음
- 자음/모음 데이터만으로 다양한 글자 조합 가능
- 쌍자음과 겹모음을 별도 라벨로 수집하지 않아도 됨

한계:

- 실제 자연어 단어/문장 보정은 아직 없음
- 받침과 복잡한 음절 조합은 규칙 정확도가 중요함
- 동적 수어 단어 인식과는 별도 문제임

## 21. 발표 대본 초안

### 문제 정의 대본

저희 프로젝트는 수어 사용자와 비수어 사용자 사이의 소통 장벽을 줄이는 것을 목표로 합니다.
비수어 사용자가 수어를 모르는 상황에서는 수어 사용자의 의도를 즉시 이해하기 어렵습니다.
그래서 웹캠으로 수어 손모양을 인식하고, 이를 텍스트 자막으로 변환하는 시스템을 만들었습니다.

### 모델 설명 대본

모델 입력으로 원본 이미지를 직접 사용하지 않고 MediaPipe Hands에서 추출한 손 랜드마크를 사용했습니다.
각 손모양은 83차원 특징 벡터로 변환되며, 이 벡터를 SVM 모델에 입력해 자음, 모음, DELETE 중 하나로
분류합니다. 이번 데이터셋은 이미지 원본이 아니라 83차원 구조화 특징을 사용하고, 라벨별 537프레임씩
균형 있게 수집되어 있습니다. 이런 조건에서는 CNN처럼 큰 이미지 모델을 쓰기보다 SVM으로 먼저
검증하는 것이 구현 비용 대비 효율적이라고 판단했습니다. 실제 평가에서도 3,456개 테스트 프레임 중
3,449개를 맞췄습니다.

### 성능 설명 대본

총 17,184프레임의 데이터를 수집했고, 32개 라벨을 학습했습니다. 테스트 데이터는 3,456프레임이며,
이 중 3,449프레임을 맞추고 7프레임을 틀렸습니다. 그래서 Accuracy는 0.9980, Macro F1-score는
0.9980이 나왔습니다. 평균 SVM 추론 시간은 0.5294ms/frame으로, 30FPS 기준 한 프레임 예산인
33.3ms의 약 1.6%입니다. 따라서 현재 측정 기준에서 SVM 분류 단계 자체는 실시간 처리 병목으로 보기
어렵습니다.
다만 실제 웹캠 환경은 조명과 사용자 차이가 있기 때문에 추가 검증이 필요합니다.

### 한계 설명 대본

현재 프로젝트는 정적 수어에 집중한 MVP입니다. 동적 수어, 실제 WebRTC 영상통화, TTS 기능은 아직
완성 범위에 포함하지 않았습니다. 앞으로는 다양한 사용자 데이터를 추가하고, 동적 수어 인식과 실제
영상통화 기능을 확장하는 방향으로 개선할 수 있습니다.
